## What this document is

This is the formal specification for the TOMB Hardware Abstraction Layer (HAL).

It defines exactly four interface methods. These four method signatures are
frozen permanently. They will never change. Any software that calls these
methods will continue to work with any future TOMB implementation.

This is the TCP/IP property. TCP/IP defined how to send packets in 1974.
It still works today. Every device on the internet speaks the same protocol.
TOMB defines how to encode data for molecular storage in 2026.
It will still work in 2126.

## The four methods

### Method 1: write

```
Input:
  strands:  StrandSet      — set of synthesis-ready oligonucleotide strands
  address:  HilbertAddress — 40-bit Hilbert space address for this data block

Output:
  WriteReceipt             — confirmation with timestamp and strand count

Guarantees:
  - All strands in input satisfy synthesis constraints before this call
  - GC content of every strand: 40–60%
  - No homopolymer run longer than 4 bases in any strand
  - Toehold at positions 25-34 is accessible (not buried in hairpin)
  - Returns error if hardware is unavailable
  - Returns error if pool capacity is exceeded

Error conditions:
  HardwareError    — hardware unavailable or synthesis failed
  ValidationError  — strand fails synthesis constraints (should not happen
                     if MolecularCompiler was used upstream)
```

### Method 2: read

```
Input:
  address:  HilbertAddress — address of the data block to retrieve
  pool:     PoolHandle     — reference to the molecular pool to read from

Output:
  Vec<DnaStrand>           — raw strands from the pool (may contain errors)

Guarantees:
  - Returns strands matching the toehold at the given address
  - May return corrupted strands (ECC layer corrects them)
  - May return fewer strands than written (strand loss is normal)
  - Returns error if address not found in pool
  - Returns error if pool is inaccessible

Error conditions:
  HardwareError    — pool unavailable or sequencing failed
  AddressError     — address not found in pool
```

### Method 3: recognise

```
Input:
  strand:   DnaStrand      — a single oligonucleotide strand

Output:
  HilbertAddress           — the address encoded in the strand's toehold

Guarantees:
  - Deterministic: same strand always returns same address
  - Reads toehold region (positions 25-34 of the 196nt strand)
  - Returns error if toehold is unreadable or corrupted beyond recovery
  - Does NOT require a pool reference — works on a single strand

Error conditions:
  AddressError     — toehold is corrupted or unreadable
  HardwareError    — strand is malformed
```

### Method 4: verify

```
Input:
  strand:   DnaStrand      — a single oligonucleotide strand

Output:
  ValidationResult         — pass/fail with details for each constraint

Guarantees:
  - Deterministic: same strand always returns same result
  - Checks GC content (must be 40–60%)
  - Checks homopolymer runs (no run longer than 4)
  - Checks strand length (must be exactly 196nt for v1.0)
  - Checks toehold melting temperature (must be 38–50°C)
  - Checks TTTT spacer at positions 21-24
  - Never throws an exception — always returns a ValidationResult

Error conditions:
  None — this method always succeeds (returns ValidationResult.valid=false
  for invalid strands rather than throwing an error)
```

## The strand structure (v1.0)

Every strand generated by TOMB v1.0 has this exact structure:

```
Position  1-20:   LEFT PRIMER (20nt)
  Positions 19-20: VERSION FIELD
    AA = v1.0, AC = v1.1, AG = v1.2, CA = v2.0, GA = v3.0, TA = v4.0

Position 21-24:   TTTT SPACER (4nt)
  Always exactly TTTT.
  Physically separates primer from toehold.
  Prevents toehold burial in hairpin secondary structure.
  MUST NOT be changed. MUST NOT be removed.

Position 25-34:   TOEHOLD (10nt)
  Encodes the Hilbert address of this strand.
  Probe = 14nt reverse complement of this region (extended 4nt into payload).
  ViennaRNA validation: positions 25-34 MUST be unpaired in MFE structure.
  Thermodynamic constraint: Tm 38–50°C.

Position 35-38:   SHABDA (4nt)
  Self-describing metadata symbol.
  Encodes content type, compression algorithm, and ECC scheme in use.
  Allows cold readers to identify strand contents without external metadata.

Position 39-176:  PAYLOAD (138nt)
  Topological symbols + lattice parity ECC.
  Scrambled with XOR using Hilbert-address-seeded LCG for GC balance.

Position 177-196: RIGHT PRIMER (20nt)
  For PCR amplification (optional — toehold retrieval does not require PCR).

Total: 196 nucleotides per strand.
```

## Backwards compatibility guarantee

A TOMB decoder implementing version M.m MUST:
1. Correctly decode all strands at version M.x where x ≤ m
2. Correctly decode all strands at version (M-1).x for all x
3. Attempt to decode strands at version (M+1).x, returning a warning
4. Never silently corrupt data — return an error if decoding fails

This guarantee means:
- Strands written today (v1.0) will be readable by any future TOMB decoder
- A library written in 2060 will decode archives written in 2026
- This is how you build a 1000-year archive that is actually readable

## What implementors must do

To build a TOMB-compatible implementation:

1. Implement the TombHardware trait (4 methods above)
2. Run the TOMB conformance test suite (tests/conformance.rs)
3. All 17 tests must pass (C-01 to C-17)
4. Contact rakshith.tomb@gmail.com with your implementation results.

That is all. No certification, no fee, no approval required.
If all 17 tests pass, you are TOMB-compatible.

---

## What implementors must NOT do

1. Change the 4 method signatures
2. Change the strand structure (positions, lengths, spacer)
3. Claim TOMB compatibility without passing the conformance suite
4. Modify the version encoding scheme
